Getting the Code and Playing with it!

Environment..

We suggest to use the Eclipse IDE as development platform (e.g., Eclipse IDE for Java Developers). You may use another package with Java development functionality.

Getting the Code..

We provide a stable version of the library as an Eclipse project (ZIP file), which includes the source files. In addition we provide a JAR file, which you can more easily use within your Java application. You can download the files from the PrimeLife server. The simplest way to start exploring the Identity Mixer library is to use the ZIP file and import it into Eclipse as an ''existing project'' into you Eclipse workspace.

Running the Test Cases..

First you have to change a URI such that it points to the required files residing on your file system. To do this you will have to change the parameter BASE_LOCATION that resides in com.ibm.zrl.idmx.tests.idmx.TestIssuance. You will have make it point to the location where the parameters (e.g., system and group parameters) are located. On Windows this will have to be something like file:///C:/Users/.../workspace/com.ibm.zrl.idmx.2_3_2/tests/com/ibm/zrl/idmx/tests/files/parameter/ on a UNIX based system the URI will rather be file:///home/username/../parameter/. Note that you will not have to change the BASE_ID as this must comply with the URI used in the example files.

Now you should be ready to have a go with the issuance test cases. Go to the TestIssuance class and run all test cases. Running the library for the first time will make the system generate the required parameters. Thus, there are a number of error messages, which you can ignore. However, it is essential that the test cases verify! If they do not verify correctly, you will have to fix your system setup before you can continue! If the issuance process works correctly you should find several Credential_n.xml files in the folder tests/files/credentials.

Having the credentials allows you to run the proof test cases - theoretically. However, there are some details that you have to take care of first.

  • If a new issuer key pair has been generated, those new bases have to go into several proof specifications. First, the proof specification ProofSpecRep_Cred2Cred3.xml specifies bases number 6, 5 and 4 of the issuer public key to used. Thus, you have to replace the values in the proof specification with the corresponding bases stored in ipk.xml. Second, the complete credential also makes use of representations. More specifically, it specifies bases number 0 and 1 to be used. That is why you have to replace the values in ProofSpec_CredComplete with the base number 0 and 1 from ipk.xml.

Now you can run the test cases of the proving protocol. Note that after running all proof test cases you can run any test individually (if it is a verification test, it will simply use the *_proof.xml and *_nonce.xml files serialised during the previous run of the proving protocol).

The last step is to run the tests from TestUpdate. Those test will update values from several credentials. Note that some credentials expire after a certain time. They MUST be updated to verify correctly again. We do not guearantee to implement updates for all credentials. Therefore you might have to re-issue selective credentials if they expire.

Understanding what is going on!

Architectural Ideas..

The Identity Mixer anonymous credential system is documented in various publications. We maintain the blog idemix@wordpress to keep collect all necessary information.

When it comes to documentation of the current code, there is a document describing the language that we use (Mixing Identities with Ease). In addition, we maintain a specification of the implemented library, which is distributed together with the code. As by now you have the test cases running, the best way of understanding the code is probably by starting a simple test case in debugging mode and follow the commands executed when running the code.

Notes!

Logging..

We use the standard Java (since Java 1.4) logging mechanism. We mainly use the following levels:

  • Level.INFO: General information.
  • Level.FINE: Detailed debuging information.
  • Level.SEVERE: Error information.

Debugging Mode..

When running the test cases or otherwise debugging the library, there is a field DEVELOPING in the class com.ibm.zrl.idmx.utils.Constants. Setting this field to true fixes the seed of the random generation mechanism - thus you can run the tests several times using the same randomness values.
